The Problem

You start in chat, ask a few questions, and everything feels cheap.

Then you switch to Claude Code, Codex, or an agent workflow. Usage spikes, responses slow down, and limits appear faster than expected.

The work did not feel 10x bigger, but the token model changed.

The Real Reason You’re Hitting Limits

Token usage = Context Size x Iteration Count

Context Size

How much text the model processes per step: files, logs, instructions, and tool output.

Iteration Count

How many times you rerun the loop: analyze, edit, validate, retry.

Core operating principle: token usage compounds when context and retries grow together.

Chat vs Code

Mode Token Usage Why
Chat Low Scoped, single-pass
Code High Context + iteration
Agents Extreme Recursive workflows

Workflow Shape

Chat (Fast Path)

User→Model→Response

Coding (Full Execution Path)

User→Model→Load Files→Analyze Code→Generate Patch→Run Tests→Process Logs→Response

Chat is a single-pass flow. Coding introduces multiple tool-driven steps and extra context.

Takeaway: LLMs are cheap for thinking, but expensive for execution.

What’s Actually Burning Your Tokens

  • loading large files or many files
  • repeated plan-edit-validate loops
  • hidden system/tool overhead
  • retries caused by ambiguous prompts

A realistic coding step:

  • context load: 15k-30k tokens
  • reasoning/planning: 5k-10k tokens
  • edit + validation: 5k-10k tokens

Total per step: 25k-50k tokens.

Run that loop 4-6 times and you are in 100k-250k+ quickly.

The Biggest Mistakes People Make

  • dumping the full repo for a local change
  • using vague prompts
  • doing exploration inside code tools
  • rerunning full pipelines for one-stage edits
  • letting agents run without hard boundaries

The Fix — Practical Playbook

Use this as an execution sequence, not a checklist you apply randomly.

  1. Separate thinking from execution. Decide approach in chat, then execute once with concrete scope in code tools.
  2. Control context aggressively. Pass only relevant files/functions/logs so each call starts from a smaller base cost.
  3. Reduce iteration count. Set clear success criteria up front so you avoid ambiguous retries.
  4. Use the right tool for each phase. Chat for planning, code tools for patching, local tooling for deterministic validation.

Operator tip: If you cannot describe the exact target file/function, you are still in planning mode, not execution mode.

A Better Workflow

  1. Plan in chat
  2. Execute in code tools
  3. Avoid unnecessary loops

Impact: this separation alone can reduce usage by 5-10x.

Real Example (Before vs After)

Naive Workflow
  • load broad repo context
  • explore architecture while patching
  • rerun full checks on each attempt
  • keep adding logs/files each retry

Estimated token impact: ~200k per task cycle

Optimized Workflow
  • plan once in chat before execution
  • send only relevant files/functions
  • run targeted checks first, full checks last
  • reuse concise summaries between steps

Estimated token impact: ~15k-40k per task cycle

Typical savings are 5-10x when scope and loop count are controlled.

Context Control Techniques

  • Pass only required files, functions, and logs.
  • Summarize non-critical context before handoff.
  • Avoid full repository ingestion for localized edits.
  • Keep instruction files concise to avoid overhead.

Pipeline Optimization

  • Break workflows into stages and rerun only changed stages.
  • Avoid end-to-end reruns for local fixes.
  • Cache stable intermediate artifacts.

Example stage split: extraction, classification, field extraction, publishing.

Rule: if one stage changes, rerun from that stage forward only.

Prompt Efficiency Patterns

Bad 
Improve this code.
Good 
Update function X to handle edge case Y.
Do not modify other parts.

Signs You’re About to Hit Limits

  • responses slow down each turn
  • repeated retries on the same task
  • larger inputs on each step
  • repeated runs of near-identical workflows

Agent Token Explosion

Pressure Points

  • recursive calls stack context
  • multi-step flows add overhead per handoff
  • persistent memory inflates prompt size

Controls

  • set hard step boundaries and stop conditions
  • cache intermediate outputs
  • cap recursion depth and retry count
  • summarize or reset memory between phases

Claude-Specific Optimization

Claude performance usually degrades for two reasons: repeated instruction overhead and bloated context payloads. Use CLAUDE.md, Skills, and MCP together to fix both.

CLAUDE.md

What it is

A stable project instruction file that encodes persistent behavior rules for Claude.

What problem it solves

Prevents repeating baseline instructions every session.

When to use

Long-lived projects with shared standards.

When not to use

Temporary, task-specific requirements that change every run.

How it helps token optimization

Reduces repeated setup prompt tokens and lowers retry rate through consistent behavior.

Common mistakes

  • turning it into a giant policy dump
  • including unstable task details
  • duplicating tool output instructions

Practical example

Define output style, testing expectations, and file-edit boundaries once instead of restating each session.

Visual 1 — CLAUDE.md

User Workflow↓CLAUDE.md↓Consistent Claude Behavior

CLAUDE.md provides stable, reusable project instructions so you do not restate them every session.

What should go into CLAUDE.md

  • project purpose
  • stable rules and boundaries
  • coding conventions
  • workflow preferences
  • things to avoid

What should NOT go into CLAUDE.md

  • task-specific one-off instructions
  • large copied reference docs
  • volatile issue-by-issue context

Suggested CLAUDE.md snippet

# Project Purpose
Build and maintain token-efficient LLM workflows.

# Stable Rules / Boundaries
- Prefer scoped context over full-repo context.
- Validate only impacted components before full runs.

# Coding Conventions
- Keep changes minimal and testable.
- Use explicit acceptance criteria in prompts.

# Workflow Preferences
- Plan in chat, execute in code tool.

# Things to Avoid
- Broad exploratory prompts during execution.
- Re-running full workflows without stage-level need.

Skills

What it is

Reusable task patterns that package instructions for repeated workflows.

What problem it solves

Reduces prompt repetition and inconsistent outputs across repeated tasks.

When to use

Recurring operations like extraction, review, refactor, and summarization.

When not to use

Ad hoc tasks that are unique and unlikely to repeat.

How it helps token optimization

Shorter prompts + lower retries because expected structure is pre-defined.

Common mistakes

  • overloading one skill with unrelated task types
  • keeping skill outputs underspecified
  • using skills where external retrieval is the real bottleneck

Practical example

Create separate skills for extraction, code review, and refactor suggestions instead of one giant generic skill.

When Skill is better than CLAUDE.md: Use CLAUDE.md for stable global behavior. Use Skills for recurring task-specific execution patterns.

MCP

What it is

An access layer for tools and targeted retrieval from files, databases, APIs, and local systems.

What problem it solves

Prevents dumping large context blocks into prompts when precise retrieval is possible.

When to use

When required context can be fetched on demand from external systems or local tooling.

When not to use

When the task needs only short static context already available in prompt scope.

How it helps token optimization

Replaces broad prompt context with targeted tool results, reducing context window pressure.

Common mistakes

  • returning oversized tool payloads without filtering
  • calling tools repeatedly without caching results
  • using MCP for logic that should be encoded in a skill

Practical example

Query only the exact table rows or log slices needed instead of pasting full datasets into the prompt.

Operating Rules

  1. Never send more context than needed.
  2. Never rerun full workflows unnecessarily.
  3. Separate planning from execution.
  4. Scope prompts tightly.
  5. Cache intermediate results.

Final Principle

LLM usage is not about how much you use. It is about how efficiently you structure the work.
