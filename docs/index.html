The Problem

Most teams start with chat and feel fine. Then coding sessions begin, context gets larger, retries accumulate, and usage limits appear unexpectedly fast.

It is usually not a model issue. It is a workflow shape issue.

Warning: Repeated broad-context retries are the fastest path to token exhaustion.

The Real Reason

Token usage = Context Size x Iteration Count.

  • Context size = how much data you pass per request.
  • Iteration count = how many times you rerun the loop.

Concrete example: Loading a full repo (200k tokens) and iterating 10 times = 2M tokens burned.

Why this matters: Most users optimize wording. The bigger win is reducing context volume and loop count.

Chat vs Code

Choose the right mode before you spend tokens: use chat for thinking, code tools for execution.

Chat (Exploration Mode)

Use chat to explore ambiguity, frame the problem, and design solution options before execution.

When to use

  • unclear requirements
  • comparing multiple approaches
  • designing architecture or prompts

Example: "Design a contract extraction pipeline with fallback logic"

Insight: Keep chat broad for thinking and tradeoffs. Then hand off a scoped plan to execution mode.
Code (Execution Mode)

Use code mode for precise execution, targeted changes, and validating outcomes against clear criteria.

When to use

  • implementing specific changes
  • debugging known issues
  • running targeted updates

Example: "Update M2 classification logic to include FX fee detection"

Insight: Code mode needs Precise instructions up front. It should produce Deterministic output with minimal retries.

What’s Burning Your Tokens

  • Full file reloads
    Why: entire files are re-sent each iteration.
    Impact: exponential token growth.
  • Repeated loops
    Why: retrying full workflows instead of partial fixes.
    Impact: duplicated context and wasted model calls.
  • Tool and log payload bloat
    Why: large outputs are pasted back into prompts without filtering.
    Impact: context windows fill with low-signal data.
Info: The biggest lever is not output length; it is reducing repeated large-input turns.

Biggest Mistakes

  • thinking in code mode instead of planning first
  • loading full repos for local edits
  • rerunning whole workflows after small changes

The Fix — Practical Playbook

  1. Separate thinking and execution.
  2. Scope context tightly before each run.
  3. Define done criteria to reduce retries.
  4. Use chat, code tools, and local validation in sequence.
Tip: Aim for one focused execution pass per problem slice.

Claude-Specific Optimization

CLAUDE.md

  • What it is: persistent instruction file for stable project behavior.
  • Why it matters: avoids repeating baseline prompt instructions each session.
  • Best practice: structure it around rules, scope, and constraints.

Example: define review format, safety boundaries, and expected validation steps once.

Skills

  • What they are: reusable capabilities for repeated workflows.
  • Why they matter: avoid re-explaining workflow logic in every prompt.
  • Best use: stable repeated tasks like extraction, review, and refactor.

Example: skill that consistently returns issue list, severity, and patch guidance.

MCP

  • What it is: tool integration layer for external context access.
  • Why it matters: reduces manual copy/paste of large context blocks.
  • Best use: targeted file, database, API, or local-tool retrieval.

Example: pull only relevant error traces and schema rows for the current task.

Info: Use Skills to standardize behavior; use MCP to standardize access.

Quick Token Optimization Checklist

  • Plan in chat first
  • Pass only required context
  • Avoid full repo loads for local edits
  • Use precise prompts
  • Rerun only changed stages
  • Cap retries and recursion
  • Use tools like CLAUDE.md, Skills, and MCP to reduce repetition

Final Principle

LLM usage is not about how much you use.

It is about how efficiently you structure the work and where you spend tokens in the loop.
